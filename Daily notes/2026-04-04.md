# Remarks

## Elasticsearch Optimization Session

Identified two major bottlenecks in the search query:

1. **Wildcard prefix queries** on `productName` — expensive full scan. Replaced with `edge_ngram` tokenizer on index side.
2. **Aggregations running on every request** — results rarely change. Added 60-second Redis cache for aggregation results.

Results (staging, 10k documents):

| Metric | Before | After |
|--------|--------|-------|
| p50 | 320ms | 85ms |
| p95 | 800ms | 210ms |
| p99 | 1.4s | 380ms |

## Caching Architecture Discussion

Options considered:

* **Option A**: Cache at BFF layer (HTTP response cache) — simple but coarse-grained
* **Option B**: Cache at service layer with Redis — fine-grained, invalidation control
* **Option C**: CDN edge caching for public endpoints — good for read-heavy public data

Decision: Option B for internal APIs, Option C for public product catalog.
